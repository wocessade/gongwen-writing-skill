# Stage G1: 结构规划 + 内容大纲 [Strategist]

**Gate:** QG1 (BLOCK)
**Goal:** 按文种模板确定章节结构、核心信息要素
**Needs Composers:** none

## Decisions

### Axis 1: 文种模板选择

按 G0 阶段确认的 `documentType` 加载 `references/document-type-templates.md` 对应模板。

```
模板加载规则:
  if documentType == 通知:
      → 加载 "通知" 模板 (通知的通用结构: 缘由→事项→要求)
  elif documentType == 报告:
      → 加载 "报告" 模板 (报告的通用结构: 情况→分析→建议)
  elif documentType == 请示:
      → 加载 "请示" 模板 (请示的结构: 理由→事项→恳请批复)
  elif documentType == 批复:
      → 加载 "批复" 模板 (批复的结构: 来文引述→答复意见→要求)
  elif documentType == 函:
      → 加载 "函" 模板 (函的结构: 缘由→事项→商洽请求)
  elif documentType == 纪要:
      → 加载 "纪要" 模板 (纪要的结构: 会议概况→议定事项→出席信息)
  elif documentType in [通报, 通告, 公告, 公报]:
      → 加载对应模板
  elif documentType in [决定, 决议]:
      → 加载对应模板
  elif documentType == 命令:
      → 加载 "命令" 模板
  elif documentType == 意见:
      → 加载 "意见" 模板
  else:
      → STOP-AND-ASK — 文种无可用模板，询问用户结构需求
```

### Axis 2: 附件规划

```
是否需要附件?
  if 正文引用外部材料 (e.g., "详见附件"):
      → 确认附件名称与顺序
      → 在正文中标注附件引用位置
  elif 文种惯例需要附件 (e.g., 转发性通知的印发原件):
      → 自动添加标准附件
  else:
      → 无需附件

附件规划格式:
  - 附件1: {附件名称}
  - 附件2: {附件名称}
  (直接在正文附件标记处按顺序列出)
```

### Axis 3: 上行文特殊处理

```
if documentType in [报告, 请示]:
    → 签发人: 使用 G0 阶段确认的签发人
    → 附注(联系人): 确认联系人姓名和联系电话
    → 结构注意:
        - 请示: 一文一事，不得多头主送
        - 报告: 不得夹带请示事项
elif documentType == 函:
    → 确认是否需标注签发人 (视机关内部规定)
```

### Axis 4: 每节核心信息点规划

列出每节需要填入的核心信息要素，作为 G2 起草阶段的填充指引。

```
核心信息点规划格式:
  1. 标题: {已确认的标题主题}
  2. 主送机关: {已确认的主送机关}
  3. 正文:
     - 段落1 (缘由/背景): {需用户提供的信息点}
     - 段落2 (主体事项): {需用户提供的信息点}
     - 段落3 (结尾/要求): {需用户提供的信息点}
  4. 发文机关: {已确认}
  5. 成文日期: {明确或占位}
  6. 附件: {如有}
```

### Axis 5: 内容信息收集

```
if 用户已提供完整正文内容:
    → 直接按模板结构化组织
elif 用户只提供了要点:
    → 按模板填充要点，标注需补充的信息
    → 如果缺少关键信息 → STOP-AND-ASK 询问
elif 用户未提供任何正文内容:
    → 按模板生成大纲框架
    → 逐节询问核心信息
    → 记录用户回答作为 G2 输入
```

## Composer Sequence

1. **加载对应模板** — 从 `references/document-type-templates.md` 加载与 `documentType` 匹配的模板
2. **规划章节结构** — 按模板确定标题、主送、正文段落、附件、发文机关、成文日期的结构
3. **列出每节核心信息点** — 为每节列出需填充的具体信息要素，标记已收集和待收集

## Gate

### QG1 Gate Checklist

- [ ] 章节结构符合文种规范
- [ ] 所有必备元素已列入 (标题、主送、正文、发文机关、成文日期)
- [ ] 附件 (如有) 已规划
- [ ] 签发人 (上行文) 已标注
- [ ] 每节核心信息点已列出
- [ ] 缺少的信息已标记并询问用户

### Gate Failure Route

```
if 章节结构不符合文种规范:
    → BLOCK — 重新加载标准模板
    → 按模板修正结构后继续
if 必备元素缺失:
    → BLOCK — 补充缺失元素
    → 通知用户补充的内容
if 上行文签发人未标注:
    → BLOCK — 要求 G0 阶段补充签发人
if 缺少关键信息且未标记:
    → BLOCK — 补全信息需求标记
```

**To Next Stage:** 文种模板选择、章节结构大纲、附件规划、核心信息点列表、签发人(如有) → G2
